## 1. 理论 TPS 计算

### 1.1 基础参数

```rust
区块 Gas 限制：30,000,000
简单转账 Gas: 21,000
出块时间：10 秒
```

### 1.2 计算公式

```
每区块交易数 = 区块 Gas 限制 / 单交易 Gas
           = 30,000,000 / 21,000
           = 1,428 交易/区块

理论 TPS = 每区块交易数 / 出块时间
         = 1,428 / 10
         = 142.8 TPS
```

### 1.3 不同交易类型的 TPS

| 交易类型 | Gas 消耗 | 每区块交易数 | TPS |
|----------|----------|-------------|-----|
| 简单转账 | 21,000 | 1,428 | 142 |
| ERC20 转账 | 65,000 | 461 | 46 |
| 合约部署 | 500,000 | 60 | 6 |
| DeFi _swap | 200,000 | 150 | 15 |
| NFT 铸造 | 150,000 | 200 | 20 |

## 2. 实际性能瓶颈

### 2.1 当前实现瓶颈

```
┌─────────────────────────────────────────────────────────────┐
│                    交易处理流水线                          │
├─────────────────────────────────────────────────────────────┤
│  1. 签名验证    →  0.5ms/交易  →  瓶颈：单线程验证         │
│  2. 交易池排序  →  0.1ms/交易  →  瓶颈：锁竞争             │
│  3. 状态读取    →  0.2ms/交易  →  瓶颈：磁盘 I/O           │
│  4. EVM 执行     →  0.5-5ms/交易 → 瓶颈：单线程执行        │
│  5. 状态写入    →  0.3ms/交易  →  瓶颈：MPT 更新           │
│  6. 区块广播    →  100-300ms   →  瓶颈：P2P 网络           │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 性能分析

| 阶段 | 耗时 | 占比 | 可并行化 |
|------|------|------|----------|
| 签名验证 | 500ms (1000 交易) | 35% | ✅ 完全可并行 |
| 交易池管理 | 100ms | 7% | ⚠️ 部分可并行 |
| 状态读取 | 200ms | 14% | ⚠️ 受 I/O 限制 |
| EVM 执行 | 500ms | 35% | ⚠️ 依赖状态 |
| 状态写入 | 300ms | 21% | ❌ 顺序执行 |
| 网络广播 | 200ms | 14% | ✅ 异步 |

**总处理时间**: ~1.8 秒/区块 (1000 交易)
**实际 TPS**: 1000 / 10 = **100 TPS**

## 3. 优化方案

### 3.1 短期优化 (v1.1)

#### 3.1.1 并行签名验证

```rust
// 当前：串行验证
for tx in transactions {
    tx.verify_signature()?;  // 0.5ms * 1000 = 500ms
}

// 优化：并行验证
transactions.par_iter()
    .try_for_each(|tx| tx.verify_signature())?;  // 500ms / 8 核 = 62ms
```

**预期提升**: 8 倍 (单线程 → 8 核并行)
**TPS 提升**: 100 → 400 TPS

#### 3.1.2 批量状态读写

```rust
// 当前：单次读写
for tx in transactions {
    let balance = state.get_balance(tx.sender)?;
    state.set_balance(tx.sender, new_balance)?;
}

// 优化：批量操作
let balances = state.batch_get_balance(senders)?;
state.batch_set_balance(updates)?;
```

**预期提升**: 5 倍
**TPS 提升**: 400 → 600 TPS

### 3.2 中期优化 (v2.0)

#### 3.2.1 并行交易执行

利用 UTXO 账户模型：

```rust
// 独立 UTXO 交易可并行执行
let results: Vec<_> = utxo_transactions
    .par_iter()
    .map(|tx| execute_utxo_tx(tx))
    .collect();
```

**预期提升**: 5-10 倍 (取决于交易独立性)
**TPS 提升**: 600 → 2,000 TPS

#### 3.2.2 流水线处理

```
区块 N    : 验证 → 执行 → 写入
区块 N+1  :       验证 → 执行 → 写入
区块 N+2  :             验证 → 执行 → 写入
```

**预期提升**: 3 倍 (流水线并行)
**TPS 提升**: 2,000 → 3,000 TPS

### 3.3 长期优化 (v3.0)

#### 3.3.1 状态分片

```
分片 0: 地址 0x0000... - 0x3FFF...
分片 1: 地址 0x4000... - 0x7FFF...
分片 2: 地址 0x8000... - 0xBFFF...
分片 3: 地址 0xC000... - 0xFFFF...
```

**预期提升**: 分片数倍 (4 分片 = 4 倍)
**TPS 提升**: 3,000 → 12,000 TPS

#### 3.3.2 Layer 2 扩展

| 方案 | TPS | 最终性 |
|------|-----|--------|
| 状态通道 | 10,000+ | 即时 |
| Rollup | 2,000-5,000 | ~1 小时 |
| Plasma | 1,000-3,000 | ~10 分钟 |
