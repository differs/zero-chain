# UTXO Compute Yellowpaper v1.1
## 去中心化计算操作系统的协议内核

**版本**：v1.1 Draft  
**时间**：2026-03  
**定位**：面向协议开发者、系统研究者、基础设施工程师

---

## 摘要

UTXO Compute v1.1 提出一种面向多域执行、可验证计算和对象级并行的协议模型。它从传统 UTXO 的原子性、安全性与自托管特性出发，将链上状态统一建模为**版本化资源对象**：每个对象的具体状态以不可变 UTXO 版本承载，而对象本身则通过稳定的 `object_id` 跨越多个版本延续。

与账户模型相比，UTXO Compute 不依赖共享全局可变状态；与传统支付型 UTXO 相比，它进一步支持状态演化、逻辑验证、显式读集、多域扩展、代理自动执行与跨域票据结算。主链负责最终性、治理和跨域锚定；应用域负责高频业务执行；通道与实例层负责实时交互。该结构使协议能够原生面向 DeFi、游戏世界、自治代理、跨域资产流转与未来的去中心化计算市场。

v1.1 的关键改进是：将“可更新 UTXO”的模糊语义收敛为**不可变输出 + 版本化对象**，并以显式依赖、票据化跨域、确定性虚拟机与协议内调度器确保系统在工程上可实现、在共识上可验证、在语义上自洽。

## 1. 引言

区块链架构长期沿两条路线发展。其一是以比特币为代表的 UTXO 模型，它强调原子性、可验证性、局部状态与并行潜力；其二是以太坊为代表的账户模型，它强调全局共享状态与通用合约执行。二者各有优势，但也各有边界。UTXO 模型在处理复杂状态机和持续演化对象时表达受限；账户模型则在高并发场景下受到共享状态竞争与串行执行约束。

UTXO Compute 的目标不是简单“在 UTXO 上叠加智能合约”，也不是“做一条兼容 EVM 的新链”，而是重新定义 UTXO：将其从账本中的价值输出，扩展为链上统一的资源对象与计算单元。资产、数据、状态、模型、订单、身份、游戏对象、代理、跨域票据，都以统一对象模型承载。对象的每一次状态变化都通过新 UTXO 版本表达，从而在不破坏 UTXO 不可变性的前提下，获得复杂状态系统所需的持续演化能力。

该协议的长期愿景是成为**去中心化计算操作系统**的信任与结算内核：主链负责最终性和锚定，应用域负责高频执行，通道层负责实时交互，代理负责自动化动作，用户在上层以意图和对象进行交互，而底层仍保有 UTXO 的安全性、并行性和自托管本质。

## 2. 设计目标与原则

### 2.1 设计目标

UTXO Compute v1.1 旨在同时满足以下目标：

1. **原子性**：任意状态转换要么全部成功，要么全部失败。
2. **对象持久性**：允许角色、仓位、池子、代理、仓库等长期存在并演化。
3. **并行执行**：冲突边界明确，允许对象级并行验证与执行。
4. **多域扩展**：支持主链、应用域与实例/通道层分层协同。
5. **统一资源抽象**：资产、数据、引用、票据、任务统一表达。
6. **自动执行**：代理能够基于规则与触发条件自主运行。
7. **跨域安全**：跨域动作通过锚定与票据完成，而非隐式共享状态。
8. **确定性**：所有节点对同一输入必须得到相同结果。

### 2.2 设计原则

#### 2.2.1 输出不可变

每个具体 UTXO 输出一经创建即不可修改。任何“更新”都必须消费旧版本并创建新版本。

#### 2.2.2 对象与版本分离

协议区分：

- **输出版本**：物理层不可变单元
- **逻辑对象**：跨多个版本延续的稳定身份

#### 2.2.3 显式依赖

任何逻辑脚本读取但不消费的对象必须进入交易 `read_set`。协议不允许未声明读取。

#### 2.2.4 验证优先

脚本的职责是验证状态转换是否合法，而不是在执行时任意重写交易。

#### 2.2.5 分层扩展

高频交互不应全部压在主链上。主链负责最终性与结算，应用域和通道负责吞吐与实时性。

## 3. 核心概念

### 3.1 版本化资源对象

UTXO Compute v1.1 的基本单元是**对象版本**。一个对象版本是一个不可变 UTXO，包含：

- 稳定对象标识 `object_id`
- 具体版本输出标识 `output_id`
- 版本号 `version`
- 所属域 `domain`
- 资源集合 `resources`
- 所有权脚本 `lock`
- 行为逻辑脚本 `logic`
- 状态 `state` 或状态承诺 `state_root`
- 生命周期参数（TTL、租金储备）
- 前驱引用 `predecessor`

同一逻辑对象在不同时间点对应不同版本输出，但只能有一个活跃版本。

### 3.2 资源

协议中的“资源”不仅是币，还包括：

- 主币与代币
- NFT 与游戏道具
- 数据块与模型哈希
- 引用其他对象的关系
- 跨域票据
- 算力、存储或服务承诺

### 3.3 对象类型

典型对象包括：

- `Value`：纯价值对象
- `State`：状态对象
- `AssetPolicy`：资产发行与销毁规则对象
- `Agent`：自治代理对象
- `Anchor`：域锚定对象
- `Ticket`：跨域票据对象
- `Channel`：通道对象
- `GameObject`：游戏对象
- `Identity`：身份与信誉对象

### 3.4 域

协议将网络划分为多个域：

- **Domain 0**：主链，负责最终结算、治理和跨域锚定
- **Application Domains**：应用域或侧链，负责高频业务执行
- **Channels / Instances**：通道与实例层，负责局部实时交互

## 4. 数据结构模型

### 4.1 标识体系

协议定义两种核心标识：

- `OutputId`：某个具体输出版本的唯一标识
- `ObjectId`：某个逻辑对象的稳定标识

这一区分解决了传统“可更新 UTXO”设计中的语义冲突问题。

### 4.2 资源映射

每个对象版本通过 `ResourceMap` 表示其持有资源。资源映射必须按 `AssetId` 排序，确保确定性序列化。资源值类型包括：

- `Amount(u128)`：同质化数量
- `Data(Vec<u8>)`：数据内容
- `Ref(ObjectId)`：对其他对象的逻辑引用
- `RefBatch(Vec<ObjectId>)`：批量引用

引用不会自动展开；若脚本需要读取被引用对象，必须显式加入 `read_set`。

### 4.3 状态存储

对象状态支持两种方式：

- **内联状态**：小型状态直接写入 `state`
- **承诺状态**：大型状态仅存 `state_root`，完整数据通过链下或证明系统提供

### 4.4 生命周期字段

每个对象版本携带：

- `created_at`
- `ttl`
- `rent_reserve`
- `flags`

这为对象租金、冻结和回收提供协议基础。

## 5. 交易模型

### 5.1 交易组成

交易由三部分构成：

- `inputs`：消费的对象版本
- `read_set`：只读但不消费的对象版本
- `outputs`：新建的对象版本草稿

此外还包括手续费、随机数和扩展元数据。

### 5.2 显式读集的必要性

在简单支付模型中，输入足以描述冲突边界；但在对象系统中，逻辑经常需要读取：

- 价格对象
- 地图状态
- 角色对象
- 预言机
- 仓库对象
- 游戏怪物或任务对象

若不显式声明，协议将无法判断并行冲突，也无法保证重放结果一致。因此 `read_set` 是 v1.1 的关键结构。

### 5.3 状态转换类型

协议支持以下转换语义：

- `Create`
- `Transfer`
- `Update`
- `Split`
- `Merge`
- `Mint`
- `Burn`
- `Anchor`
- `TicketFinalize`
- `ChannelSettle`

其中 `Update` 被严格定义为：**消费旧版本 + 创建新版本**。

## 6. 验证规则

### 6.1 基础检查

节点验证交易时，首先检查：

1. 输入对象是否存在且未花费
2. 读集对象是否存在
3. 输入与读集是否属于交易域
4. 输出格式是否合法
5. 编码是否满足规范
6. 引用集合是否无重复

### 6.2 锁脚本验证

每个输入对象必须通过 `unlock` 与对应 `lock` 的验证。该层处理：

- 所有权证明
- 多签控制
- 时间锁
- 委托权限

### 6.3 逻辑脚本验证

若输入对象携带 `logic`，则节点需要将以下上下文传入脚本：

- 当前对象版本
- 显式读集对象
- 输出草稿列表
- 当前区块上下文

脚本返回是否接受该转换。脚本不应拥有随意重写交易结构的能力；交易由发起者构造，逻辑负责验证其正确性。

### 6.4 资源守恒

默认情况下，各类资源守恒。例外情况包括：

- 主币手续费
- 经 `AssetPolicy` 授权的增发与销毁
- 经票据和锚定规则允许的跨域锁定/释放

### 6.5 活跃版本唯一性

任意时刻，同一 `object_id` 只能有一个活跃版本。节点必须拒绝对非当前活跃版本的更新。

## 7. 虚拟机与确定性执行

### 7.1 双脚本模型

协议区分两类脚本：

- `lock`：权限控制
- `logic`：状态机验证

这种结构让所有权边界和业务语义清晰分离。

### 7.2 WASM 作为主逻辑环境

WASM 是 v1.1 的主执行环境，因为其具备：

- 良好性能
- 多语言编译能力
- 沙箱化执行
- 易于定义确定性约束

### 7.3 EVM 兼容层

EVM 兼容可作为应用域能力或迁移层存在，但不应主导协议核心状态模型。UTXO Compute 的核心抽象是对象版本，而非账户存储槽。

### 7.4 确定性约束

协议必须保证：

- 固定 VM 版本
- 固定 gas schedule
- 禁止本地时间与网络访问
- 禁止未声明对象读取
- 浮点禁用或严格规范

## 8. 并行执行

### 8.1 冲突规则

两笔交易发生冲突，当满足以下任一条件时：

1. 消费同一输入对象
2. 一笔写入对象，另一笔读取或写入该对象
3. 同时更新同一逻辑对象的活跃版本

### 8.2 并行原则

无冲突交易可以并行执行。并行只是一种执行优化，不改变区块内的最终确定性顺序与语义。

### 8.3 应用意义

对象级并行为高频复杂场景提供基础。例如：

- 不同地图的玩家活动并行
- 不同副本中的战斗并行
- 不同流动性池交易并行
- 不同代理运行并行

这类域适合承载：

- DeFi 执行域
- 游戏世界域

通道与实例层适用于实时交互：

- 游戏对战

- 私有临时会话

只有开启、争议和最终结算需要返回上层域。

## 10. 锚定与跨域协议

### 10.1 锚定对象

每个域在主链上拥有锚定对象，记录：

- 最新可信状态根
- 最新可信高度
- 锚定承诺

### 10.2 票据对象

跨域动作通过票据完成。票据包含：

- 源域与目标域
- 资源承诺
- 负载哈希
- nonce
- 超时高度
- 当前状态

### 10.3 两阶段流程

1. **Prepare**：源域锁定或销毁对象，生成票据
2. **Relay / Finalize**：目标域验证票据与锚定，创建新对象
3. **Commit / Refund**：完成回执或超时退款

### 10.4 安全要求

协议必须明确：

- 锚定频率
- 最终性窗口
- 票据唯一性
- nonce 防重放
- 超时退款路径
- 域重组处理规则

## 11. 状态通道与实例层

### 11.1 通道对象

通道对象保存：

- 参与方
- 当前状态承诺
- 序列号
- 结算规则
- 超时参数

### 11.2 使用场景

通道适用于：

- 实时对战
- 点对点交互
- 高频链下操作
- AI 回合制协作

### 11.3 设计意义

这使协议的“无限扩展”不依赖单链 TPS 神话，而建立在：

- 主链结算
- 应用域高频执行
- 通道实时局部交互

之上。

## 12. 代理 UTXO

### 12.1 定义

代理是特殊状态对象，拥有：

- 调度策略
- 目标函数
- 权限
- 记忆状态
- gas 预算

### 12.2 调度

代理可由以下方式唤醒：

- 固定区块间隔
- 特定区块高度
- 条件触发
- 事件触发

### 12.3 执行

代理执行时：

- 读取自身和声明依赖
- 在 VM 中运行逻辑
- 生成内部交易
- 创建自身下一版本
- 支付执行 gas

### 12.4 场景

代理可表示：

- 套利机器人
- 自动补仓/清算器
- DAO 自动投票器
- 游戏 NPC
- 宠物与怪物
- 世界事件控制器
- 跨域路由代理

## 13. 游戏原生支持

### 13.1 为什么游戏是核心验证场景

游戏天然要求：

- 高频对象交互
- 长期状态对象
- 跨对象依赖
- 高并发
- 动态事件
- 可验证经济系统
- 自治代理参与

因此，若协议能严谨支撑游戏，就通常能支撑更一般的复杂系统。

### 13.2 游戏对象建模

在 UTXO Compute 中：

- 角色 = `GameObject`
- 装备 = `Value` / `GameObject`
- 怪物 = `GameObject + Agent`
- 地图 = `State`
- 仓库 = `State`
- 副本 = `Channel` 或实例对象
- 任务 = `Ticket` / `State`
- 宠物 = `Agent`

### 13.3 推荐分层

- 主链：身份、资产、结算
- 游戏域：世界状态与经济
- 通道/实例层：战斗、回合、实时局部交互

## 16. 与现有模块化链的差异

UTXO Compute 与多链/分片系统在架构外观上相似，但底层原语不同。

其核心差异在于：

1. 不是账户分片，而是**对象级资源系统**
2. 不只是链间并行，也支持**域内对象级并行**
3. 资源、状态、数据、票据和代理采用统一对象模型
4. 游戏、AI 与自治代理是协议原生目标，而非附加生态

## 18. 结论

UTXO Compute v1.1 通过“不可变输出 + 稳定对象标识 + 显式依赖 + 多域锚定 + 代理调度”的组合，建立了一套既严谨又具扩展性的协议内核。它保留了 UTXO 的原子性、安全性和并行性，又补足了状态演化、自动执行、跨域结算和复杂对象系统所需的能力。

该协议不应被理解为“又一条支持智能合约的 UTXO 链”，而应被理解为通往**去中心化计算操作系统**的一层基础协议：主链提供信任与最终性，应用域提供吞吐与表达能力，通道与实例层提供实时交互，代理提供自动化执行，而统一对象模型为未来的资产、游戏、AI 与自治经济提供共同基础。
