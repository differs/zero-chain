# ZeroChain 设计理念（Design Philosophy）

> 最后更新：2026-03-11  
> 交付基线（自 2026-03-07 起）：**ZeroChain 计算模型（UTXO Compute + ed25519 + `zero_*` 接口）**

本文件用于回答三个问题：

1. ZeroChain **是什么**（定位与边界）？
2. 我们 **为什么** 这样设计（核心权衡）？
3. 这些理念对实现与贡献意味着 **什么**（工程红线与约束）？

---

## 1. 定位：PoW 安全底座上的 UTXO Compute

ZeroChain 是一条以 **PoW 共识**提供无许可安全性的主链，其执行与状态表达采用 **UTXO Compute** 的原生路径：把链上状态统一为“**版本化资源对象**”，用 UTXO 的不可变性承载对象的每一次状态演化，并以显式依赖与确定性执行保证可验证性与可复现性。

我们把“协议的结算与最终性”与“高频应用执行”解耦：主链优先承担最终结算、锚定与安全边界；上层扩展（域/实例/通道/代理）在不破坏主链验证性的前提下获取吞吐与交互性能。

---

## 2. 我们追求的目标（What we optimize for）

这些目标来自协议设计的第一性原则，并贯穿实现与运维：

1. **原子性（Atomicity）**：状态转换要么全部成功，要么全部失败。
2. **对象持久性（Persistent Objects）**：长期存在的对象（仓位/池子/代理/游戏对象等）可持续演化。
3. **并行性（Parallelism）**：冲突边界清晰，支持对象级并行验证与执行。
4. **多域扩展（Multi-domain）**：主链 + 应用域/实例/通道分层协同。
5. **统一资源抽象（Unified Resources）**：资产、数据、引用、票据、任务用统一对象模型表达。
6. **自动执行（Agents / Automation）**：代理可以基于规则与触发条件自动运行。
7. **跨域安全（Cross-domain Safety）**：跨域动作通过锚定与票据结算，而不是隐式共享状态。
8. **确定性（Determinism）**：同一输入在所有节点上必须得到相同结果。

---

## 3. 核心设计原则（Non-negotiables）

### 3.1 单一路径：默认只保留 ZeroChain 计算路径

- 默认对外暴露 ZeroChain RPC 方法集：`zero_*`，并保留 `net_*` 网络探针方法与 `zero_clientVersion` / `zero_keccak256` 信息方法。
- 任何兼容逻辑都必须是**显式开关**、默认关闭，并且需要评审确认。

### 3.2 输出不可变：用“消费旧版本 + 创建新版本”表达更新

- 单个输出（UTXO）一经创建不可修改。
- 任何“更新”都必须消费旧版本并创建新版本，从而保持可验证性与可追溯性。

### 3.3 对象与版本分离：稳定身份 + 版本演化

- **版本输出**：物理层不可变单元（共识验证最小单元）。
- **逻辑对象**：跨多个版本延续的稳定身份（业务语义承载）。

### 3.4 显式依赖：读什么就声明什么

- 交易必须显式声明读集（`read_set`）：读取但不消费的对象也必须被声明。
- 禁止“隐式读取/未声明依赖”，以保证并行验证与冲突判定的可计算性。

### 3.5 验证优先：脚本负责验证，不负责“随意改写交易”

- 逻辑脚本的职责是验证状态转换是否合法。
- 对确定性与可审计性有破坏的“运行时魔法”属于反模式。

### 3.6 分层扩展：主链做结算与最终性，上层做吞吐与交互

- 主链承担：最终性、结算、安全边界、跨域锚定（以及必要的治理机制）。
- 应用域/实例/通道承担：高频业务执行与实时交互，并通过锚定/票据与主链安全对齐。

---

## 4. 工程理念：红线优先（Redline-first）

我们把“可运维、可排障、可回归”的工程属性视为协议安全的一部分：

- **Fail-fast 默认**：配置、存储、同步、协议载荷非法必须显式失败。
- **禁止 silent fallback**：不得回退到 `default` / `mem` 等继续运行。
- **禁止关键链路吞错**：不要用 warning 代替失败返回。
- **兼容逻辑必须显式开关**：默认关闭；允许例外必须 `REDLINE_ALLOW` + 原因注释。
- **修复必须有反向测试**：证明旧行为会掩盖问题，新行为会显式失败。

> 详见：`docs/ENGINEERING_REDLINES.md`

---

## 5. 非目标（What we intentionally do NOT optimize for）

为保证协议语义与工程可控性，以下方向不作为默认追求：

- **默认保留历史兼容执行路径**（如需兼容，必须以显式开关、隔离边界与可验证约束实现）。
- **“尽量不报错”的体验优先**（在核心链路上宁可失败，也不接受掩盖故障）。
- **依赖全局共享可变状态的执行模型**（会破坏并行边界与可验证性）。
- **非确定性执行**（例如依赖本地时钟/随机数/外部 I/O 的未约束行为）。

---

## 6. 这对贡献者意味着什么（Contribution checklist）

当你提交任何涉及核心链路的改动时，请用本文件做自检：

1. 是否保持当前默认计算路径？（兼容逻辑是否有显式开关且默认关闭）
2. 是否保持 **确定性**？（输入相同输出必相同）
3. 是否保持 **显式依赖**？（读集/冲突边界清晰）
4. 是否遵守 **fail-fast / no silent fallback**？
5. 是否补齐 **反向测试**，并能在回归中阻断旧行为？

仓库内的红线门禁与相关脚本：

- `bash scripts/no_silent_fallback.sh`
- CI：`.github/workflows/redline-guard.yml`
- 发布门禁：`scripts/run_tests.sh`
